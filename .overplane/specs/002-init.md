# Spec #0002 — `overplane init`

> **Status: implemented.** This document has been updated post-implementation
> to describe the **as-built** state. Amendments folded in along the way:
> silent checks during init (`check --quiet`), recursive `--dir` creation,
> dir-relative `path` values in init logs, the global `--log-timestamps`
> flag, and the inversion of the schema's canonical home to the embedded copy
> (§2.4).

This is a **self-contained build prompt** for an AI coding agent. It extends
spec `#0001` (Go CLI bootstrap): every norm established there and in
`AGENTS.md` applies unchanged — stdlib-first, exit-code table, themed
output, `serde/canonjson`/`serde/yamlcanon` for all emitted JSON/YAML,
`hashutil` for hashing, `timeutil` for time, hermetic §14.6-style tests, and
the full `make ci` / `make ci-rehearse` gate discipline. Every requirement
below is normative unless marked *advisory*.

The deliverable, entirely within **overplane** (this module):

1. The `init` command, the `overplane.yaml` schema (embedded under
   `static/files/schema/`) + loader, a command-interface rename, and an
   extension to `config validate`.
2. A generalization of the `spec-metadata.schema.json` defaults that `init`
   seeds new specs from.
3. Documentation-grade schema prose: the public reference docs for
   `overplane.yaml` and the `init` command are generated from the schema and
   the CLI manifest by a docs toolchain maintained **outside this module**
   (§7); overplane's contribution is the prose itself.

---

## 1. Goal and non-goals

**Goal.** `overplane init` turns any directory into an Overplane project:
it creates `overplane.yaml` from schema defaults, validates it strictly,
creates the configured specs and cache directories, seeds a starter spec, and
maintains `.gitignore`. Re-running is always safe: a fully initialized project
is validated and left byte-for-byte untouched.

**Non-goals.** No network calls. No interactive prompts (init is fully
non-interactive; the TUI stack is not involved). No migration/upgrade logic
for future `schema_version` values (validate and reject, don't rewrite). No
removal or alteration of the shared config files an enclosing repository may
provide (`config/data/{global,theme,infra}.yaml`, #0001 §8.1).

## 2. The project file: `overplane.yaml`

### 2.1 Initial file (normative sample)

`init` generates exactly this file (keys in `yamlcanon` sorted order; emitted
via `yamlcanon.MarshalPlainWithBanner` — a plain-scalar variant added for
human-edited files — so output is deterministic and untagged):

```yaml
# Generated by overplane init. Safe to edit; re-run `overplane init` to validate.
# yaml-language-server: $schema=https://overplane.dev/schema/overplane.schema.json
dirs:
  cache: .cache/overplane
  specs: .overplane/specs
model:
  default: ""
project:
  description: ""
  name: my-project
schema_version: 1
```

with one substitution: `project.name` is set to the **basename of the working
directory**, lowercased, with any character outside `[a-z0-9._-]` replaced by
`-` (fall back to `my-project` if the result is empty).

### 2.2 JSON Schema (inferred from the sample above)

Author `overplane.schema.json` (embedded at
`static/files/schema/overplane.schema.json`, §2.4), draft 2020-12, following
the established schema conventions (`$schema`, `$id`
`https://overplane.dev/schema/overplane.schema.json`, `title`, top-level
`description`, root `default` object, `examples`, **`additionalProperties:
false` at every object level**). This schema is rendered verbatim in the
public reference docs (§7), so **every field carries an elaborate, user-facing
`description`** — explain what the field does, what consumes it, and its
constraints in prose, not one-liners.

| Pointer | Type | Constraints | Default |
|---|---|---|---|
| `/schema_version` | integer | `const: 1` (bump per format revision) | `1` |
| `/project` | object | required | — |
| `/project/name` | string | required; 1–100 chars; pattern `^[a-z0-9][a-z0-9._-]*$` | `my-project` |
| `/project/description` | string | optional; ≤ 500 chars | `""` |
| `/dirs` | object | required | — |
| `/dirs/specs` | string | required; relative path (semantic check §2.3) | `.overplane/specs` |
| `/dirs/cache` | string | required; relative path (semantic check §2.3) | `.cache/overplane` |
| `/model` | object | optional | — |
| `/model/default` | string | optional; free-form model identifier stub for future agent features | `""` |

Required at root: `schema_version`, `project`, `dirs`. The root `default`
block must contain the complete §2.1 document (it is the generation source,
§4.3, and the docs example source, §7).

**Directory intent (must be stated in the schema descriptions):**
`dirs.specs` and `dirs.cache` are paths **relative to the project
root** (the directory holding `overplane.yaml`, i.e. the init working dir).
The specs tree (default `.overplane/specs`) is the durable, **checked-in**
home of the project's specs; the cache tree (default `.cache/overplane`) is
disposable, machine-local state that is **never committed** — which is why
init maintains the `.gitignore` entry (§4.3 phase 5).

### 2.3 Semantic validation (third pass, Go)

Beyond the schema, `project.Config.Validate()` (§3) enforces:

- `dirs.specs` and `dirs.cache` are **relative** paths: not absolute, no
  `..` segment, non-empty after `filepath.Clean`, and distinct from each
  other.
- Violations are reported as `jsonschema.Problem`s with accurate JSON
  Pointers (e.g. `/dirs/cache`) so all validation failures print uniformly.

### 2.4 Embedded copy, external mirror, drift gate

- **Embedded:** `static/files/schema/overplane.schema.json`, picked up by the
  existing asset generator (`go generate ./...` → `assets_gen.go`). The
  binary **always** validates against the embedded copy — overplane is
  self-contained and must work in any standalone checkout.
- **External mirror (optional):** an enclosing repository may maintain a
  byte-identical mirror of the schema under its own `config/schema/` (e.g. to
  feed externally maintained reference docs, §7). Overplane never reads it at
  runtime.
- **Drift gate:** a default-tier test resolves an enclosing root via
  `paths.Resolve("")`; if found, it asserts the embedded copy is
  byte-identical to the mirror, and **skips** (with an explanatory skip
  message) when no such root exists (standalone checkout). The enclosing
  repository's CI therefore enforces sync; overplane itself stays
  self-contained.

## 3. Project package: `internal/project`

New app-agnostic-adjacent package owning everything about a user project's
`overplane.yaml` (it may import `platform/*`; `platform/*` must not import
it):

```go
const FileName  = "overplane.yaml"
const SchemaURL = "https://overplane.dev/schema/overplane.schema.json" // yaml-language-server pragma target

type Config struct {  // as-built: named field types ProjectInfo/Dirs/ModelSettings
    SchemaVersion int           `json:"schema_version" yaml:"schema_version"`
    Project       ProjectInfo   `json:"project" yaml:"project"`        // Name, Description
    Dirs          Dirs          `json:"dirs" yaml:"dirs"`              // Specs, Cache
    Model         ModelSettings `json:"model,omitempty" yaml:"model"`  // Default
}

func Default(name string) (*Config, error)      // schema-default config + sanitized name (§2.1)
func SanitizeName(name string) string           // dir basename → valid project.name (§2.1)
func Locate(startDir string) (string, bool)     // upward walk for overplane.yaml; stop at fs root
func Load(path string) (*Config, error)         // read → schema validate (embedded copy) →
                                                // decode (configloader) → Config.Validate()
func (c *Config) Validate() []jsonschema.Problem // semantic checks (§2.3)
func SchemaBytes() []byte                       // embedded schema, for config validate (§6)
```

- `Load` returns `configloader.ValidationError` (or an error wrapping the
  problems) on any schema/semantic failure so callers print
  `pointer: message` lines uniformly and exit 3.
- `Default` must be **derived from the embedded schema's root `default`**
  (decode the schema's `default` block into `Config`), not duplicated as Go
  literals — single source of truth. A unit test asserts
  `Default("x")` validates cleanly against the schema.

## 4. CLI changes: `internal/cli`

### 4.1 Interface rename: `ConfiglessCommand` → `ConfigOptionalCommand`

Rename the interface (same shape — `Name() string; Usage() string;
Run(ctx, args) error`) and update all implementors, docs, and the §9.1 prose
of the architecture notes. **Semantics:** a config-optional command runs with
or without a project `overplane.yaml`; when it wants the config it calls
`project.Locate` + `project.Load` itself. A future `ConfigRequiredCommand`
(dispatcher locates/loads/validates before `Run`, hard-failing when absent)
is anticipated but **not** built in this spec. Update `AGENTS.md` §Subcommands
wording accordingly.

### 4.2 `init` command surface

| Item | Value |
|---|---|
| Registry entry | `init` — "initialize an Overplane project" |
| Usage | `overplane init [--dir PATH]` |
| Flags | `--dir PATH` — working directory to initialize (default: CWD). Parsed with stdlib `flag`; relative paths resolved against the caller's CWD. |
| Help | full `color.HelpSpec` with flag, description, and two examples (`overplane init`, `overplane init --dir ./myproject`) |
| Kind | config-optional (§4.1) |

If `--dir` points at a nonexistent path, init logs a WARN and creates the
directory recursively (`MkdirAll`), then continues as normal; an existing
non-directory path or other IO failure exits 4. A missing CWD (no `--dir`)
is never created.

### 4.3 Execution order (normative)

`init` performs these phases **in this exact order**, stopping at the first
failure:

1. **Resolve working dir** — `--dir` or CWD; abs-path it.
2. **System checks gate** — run the same checks as `overplane check` (with
   the §6.2 amended semantics) **silently**: no check table or per-check
   output during init (the standalone `check` command gains a `--quiet` flag
   exposing the same no-output mode; result is the exit code only). Gating
   rule:
   - **container engines**: at least one engine (`docker` *or* `podman`)
     must be `ok`; if none is, fail. Individual missing engines do not
     matter as long as one works.
   - any **git** result with non-`ok` status → fail;
   - **API-key** results **never block init** — neither missing nor
     malformed keys (and, like all check output, they are not printed
     during init).
   On failure: exit 6 with a one-line error naming the failed checks. Nothing
   is created when the gate fails. Factor `runChecks` so init reuses it
   without re-implementing (no copy-paste of check logic).
3. **Create `overplane.yaml` if missing** — write the §2.1 document
   (banner + `yaml-language-server` pragma via
   `yamlcanon.MarshalPlainWithBanner`, `project.Default(basename)`),
   permissions 0644. Creation is logged at
   **WARN** (§5). If the file exists, leave it untouched (never rewrite,
   reformat, or upgrade).
4. **Read, parse, and validate `overplane.yaml`** — `project.Load`. On any
   failure **die loudly**: print every problem as `pointer: message` plus a
   `hint` (e.g. "fix overplane.yaml or move it aside and re-run overplane
   init"), exit 3. All later phases consume the **validated** config — the
   directory paths come from `dirs.*`, not from constants.
5. **Cache dir + .gitignore** —
   a. `os.MkdirAll` the configured `dirs.cache` (default `.cache/overplane`)
      — WARN if any directory was actually created.
   b. Ensure the working dir's `.gitignore` contains a line exactly
      `.cache/` (the **whole** cache root is ignored, deliberately broader
      than `dirs.cache`): create `.gitignore` (0644) if absent, or append
      the line (preceded by a newline if the file does not end with one) if
      no existing line equals `.cache/` after trimming. Any content change →
      WARN. An already-compliant file is untouched.
6. **Specs dir + seed spec** —
   a. `os.MkdirAll` the configured `dirs.specs` (default `.overplane/specs`)
      — WARN if created.
   b. **Only if init freshly created the specs directory in this run** (and
      it is therefore empty), create `001-bootstrap.md` (0644, WARN) with:
      - YAML frontmatter decoded from the **generalized
        `spec-metadata.schema.json` root `default`** (§8), with two runtime
        substitutions: `title` → `<project.name> bootstrap`, `created` →
        today's date via `timeutil` (`NowUTC`, date portion);
      - a short lorem-ipsum body: an H1 matching the title, then `## Goal`,
        `## Requirements`, `## Acceptance` sections of 2–3 lorem ipsum
        sentences each — placeholder prose the user replaces.
      The seeded frontmatter must validate against
      `spec-metadata.schema.json` (unit-tested). A **pre-existing** specs
      directory — even an empty one — is never seeded: if the user emptied
      it deliberately, init must not refill it.
7. **Summary + next steps** — one INFO log line, e.g.
   `project initialized` / `project already initialized; overplane.yaml valid`
   with `path=<workdir>`, followed by a friendly themed table
   (`color.Table`, on stdout) of next commands. Rows (command in slot 4,
   plain description):

   | Next step | Description |
   |---|---|
   | `$EDITOR overplane.yaml` | review project settings |
   | `$EDITOR <dirs.specs>/001-bootstrap.md` | write your first spec (row shown only when init seeded it) |
   | `overplane check` | re-run system checks anytime |
   | `overplane config validate overplane.yaml` | validate config after edits |
   | `overplane help` | explore all commands |

   The table appears on **every** successful run (fresh and re-run), with
   the seed row included only when the seed file exists.

### 4.4 Idempotency and non-destructiveness (hard requirements)

- `init` **never** overwrites, truncates, rewrites, or deletes any existing
  file or directory. The only writes are: create-missing files, `MkdirAll`,
  and the `.gitignore` append.
- Running `init` twice in a row: the second run performs phase 2 (checks) and
  phase 4 (validation) only as observable effects, emits **zero WARN logs**,
  changes **zero bytes and zero mtimes** under the working dir, and exits 0.
  This is the "safe to run frequently" contract — a test enforces it.
- A failed run (gate or validation) leaves previously existing files exactly
  as they were.

### 4.5 Exit codes (per the #0001 table)

| Condition | Code |
|---|---|
| success (fresh or re-run) | 0 |
| usage error (`--dir` missing arg, stray args) | 2 |
| `overplane.yaml` schema/semantic validation failure | 3 |
| working dir missing / unwritable / IO failure | 4 |
| check gate failure (§4.3 phase 2) | 6 |

## 5. Logging

- **WARN** — every file or directory creation, and every modification of an
  existing file's contents (the `.gitignore` append). Structured keys per
  #0001 §5.4: `step` (`init-config`, `init-cache`, `init-gitignore`,
  `init-specs`, `init-seed-spec`), `action` (`created`, `appended`), `path`.
  `path` values are rendered **relative to the init working dir** (e.g.
  `path=overplane.yaml`); only the working dir itself (the `--dir` creation
  WARN and the summary line) is absolute.
  Future deletions (none exist in init) follow the same rule:
  content-changing operations are WARN.
- **INFO** — exactly one summary line per run (§4.3 phase 7).
- **DEBUG** — per-item no-op notes on re-runs (`action=skipped` /
  `action=validated` with `path`), so `--log-level debug` shows the full
  decision trail.
- **Presentation contract:** idempotent validations and skips must read as
  quiet background detail — DEBUG lines, which the #0001 pretty handler
  renders plain/dim and unpadded — while on-disk mutations must visually pop
  — WARN lines, rendered bright via the theme's warn slot. A user glancing at
  init output should instantly distinguish "nothing happened" (dim) from
  "files were created/changed" (bright).
- Validation failures print problems to **stderr** (they are diagnostics) and
  use the standard error path; stdout carries only the next-steps table
  (checks run silently, §4.3 phase 2).
- A global `--log-timestamps` flag (env `OVERPLANE_LOG_TIMESTAMPS`, default
  true) controls timestamp emission in both log formats;
  `--log-timestamps=false` yields deterministic, screencast-friendly output
  (the init screencast tape sets it via `Env`).

## 6. Sibling command changes

### 6.1 `config validate` extension

`overplane config validate <path>` learns the `overplane.yaml` basename: it
validates the file against the **embedded** schema (`project.SchemaBytes`),
including the §2.3 semantic pass, printing problems and exiting 0/3 exactly
like the other supported basenames. The no-argument mode is unchanged (it
still validates the discovered shared `global`/`theme`/`infra` files,
#0001 §8.1, and does **not** search for `overplane.yaml`). Update the command
help text.

### 6.2 `check` semantics amendments (supersedes #0001 §12 pass/fail rules)

The per-check probes and table/JSON output are unchanged; only the pass/fail
aggregation changes:

- **Container engines — at least one required.** The run fails only when
  **no** configured engine (`docker`, `podman`) reports `ok`. An individual
  engine that is `not installed` or `daemon unavailable` is reported with
  that status but is non-fatal as long as another engine is `ok`.
- **API keys — never required.** A missing/empty key reports status
  `warning` (detail `not set`, hint `export <NAME>`), not `missing`-as-
  failure. A key that is *set but malformed* (interior whitespace/quotes)
  remains `invalid` and **does** fail the run — a present-but-broken key is
  a real misconfiguration.
- **Git** — unchanged (warnings only, never fails `check`; init applies its
  own stricter gate, §4.3 phase 2).

Net effect: `overplane check` exits 6 iff (no engine is `ok`) or (any API
key is `invalid`). Update the check command's help text, the §12 prose
mirror in the #0001 as-built notes is *not* edited (this spec is the record),
and the `cli.overplane.json` description for `check` to match.

satisfy — the rendering work itself is owned by the docs toolchain.

### 7.1 Configuration reference — `overplane.yaml`

- The public Configuration reference renders the `overplane.yaml` schema
  verbatim: sections from schema top-level keys, examples from schema
  defaults, with `overplane.yaml` as the **sole** Configuration section.
- Reference prose (`description` fields from the schema and the CLI manifest)

- Because the page copy is generated from the schema, the "elaborate and
  filled in" requirement is satisfied **in the schema's `description`

### 7.2 Command-line reference — add `init`, move to top

- The CLI manifest (`cli.overplane.json`, maintained with the docs toolchain)
  gains the `init` command entry — usage, description, the `--dir` flag

  examples — placed **first** in the `commands` array. Final order:
  `init, version, check, config validate, theme preview, demo`.
- The manifest's top-level `description` mentions project initialization.

### 7.3 Screencast: demonstrate `overplane init`

A screencast is recorded with the docs toolchain's existing VHS pipeline and
embedded in the public docs. The recording contract:

- The tape `cd`s into a fresh temp directory (`mktemp -d`), `git init`s it
  (so the git check passes), runs `overplane init` (capturing the WARN
  creation lines and the next-steps table), then runs `overplane init` a
  second time to show the idempotent (dim) run. It sets
  `OVERPLANE_LOG_TIMESTAMPS=false` for deterministic output (§5).
- The rendered screencast is embedded on the getting-started guide step that
  introduces `init` and on the Command Line reference directly under the

The `yaml-language-server` pragma points at

the schema at that URL so editor validation resolves; if that requires more

## 11. Acceptance criteria

1. `make ci` and `make ci-rehearse` pass.
2. In a fresh empty temp dir, `overplane init` runs the checks silently, then
   creates `overplane.yaml`, `.cache/overplane/`, `.gitignore` (containing
   `.cache/`), and `.overplane/specs/001-bootstrap.md`, each creation logged
   at WARN (bright), and finishes with the next-steps table; exit 0.
3. Immediately re-running `overplane init` produces zero WARN logs, changes
   zero bytes/mtimes, prints the INFO "already initialized" summary plus the
   next-steps table, exit 0; with `--log-level debug` the per-item skips
   appear dim.
4. With a hand-broken `overplane.yaml` (unknown key, wrong type, absolute
   `dirs.cache`), `overplane init` exits 3 listing every problem with a JSON
   Pointer and a remediation hint; the file is untouched.
5. With **no** working container engine, or with git missing/broken,
   `overplane init` exits 6 and creates nothing; with exactly one working
   engine (docker *or* podman) and any number of missing API keys, init
   proceeds. `overplane check` itself passes with one engine and unset keys,
   and fails with zero engines or a malformed set key (§6.2).
6. `overplane init --dir <path>` initializes `<path>` regardless of CWD; a
   nonexistent path is created recursively after a WARN and the run continues
   normally; a path naming an existing non-directory exits 4.
7. `overplane config validate <dir>/overplane.yaml` exits 0 on the generated
   file and 3 on a broken one.
8. The generated `overplane.yaml` byte-matches the §2.1 sample (modulo
   `project.name`) including the banner and `yaml-language-server` pragma.
9. The embedded schema is byte-identical to any enclosing repository's
   mirror copy (drift test green where a mirror resolves; skipped in a
   standalone checkout, §2.4).

12. `ConfiglessCommand` no longer exists; all commands implement

## Appendix A. Clarification answers

Supplied before writing this spec; normative.

1. The initial `overplane.yaml` sample is drafted in §2.1 of this spec and is
   the source from which the JSON Schema is inferred.
2. Field groups included: `project` (name/description), `dirs`
   (specs/cache — init creates whatever these resolve to), `schema_version`,
   and a `model.default` stub. No `checks` overrides yet.
3. The schema ships embedded in overplane (`static/files/schema/`); an
   enclosing repository may keep a byte-identical mirror enforced by a CI
   drift gate (§2.4).
4. Check gating: container engines and git gate init; API-key problems print
   normally but never block. (Refined by answer 17: at least one engine —
   docker *or* podman — suffices.)
5. `ConfiglessCommand` is renamed to `ConfigOptionalCommand` with
   config-optional semantics; a dispatcher-loaded config-required interface
   is deferred.
6. Working-directory override is `--dir PATH`.
7. `.gitignore` receives the broad `.cache/` line, not `.cache/overplane/`.
8. `spec-metadata.schema.json` defaults are generalized to neutral
   placeholders before seeding from them.
9. No-op re-runs log one INFO summary plus DEBUG per-item notes; creations
   and content changes are WARN.
10. `overplane config validate` learns the `overplane.yaml` basename.
11. The public Configuration reference treats `overplane.yaml` as the
    public-facing config file and documents it as the sole Configuration
    section (§7.1).
12. Command-line reference order: `init` first, then the existing order
    (`version, check, config validate, theme preview, demo`).
13. The generated file carries a banner comment plus a
    `yaml-language-server: $schema=…` pragma pointing at the hosted schema
    URL.
14. Frontmatter: id `#0002`, parent `['#0001']`, status `active`.

Second round of amendments (also normative):

15. `dirs.specs`/`dirs.cache` are relative to the project root; `.overplane`
    is intended to be checked in, `.cache` is not (hence the `.gitignore`
    entry).
16. The skeleton `001-bootstrap.md` is seeded **only** when init freshly
