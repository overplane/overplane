# Spec #0003 — `overplane agent`

This is a **self-contained build prompt** for an AI coding agent. It extends
specs `#0001` (Go CLI bootstrap) and `#0002` (init): every norm established
there and in `AGENTS.md` applies unchanged — stdlib-first, the exit-code
table, themed output, `serde/canonjson`/`serde/yamlcanon` for all emitted
JSON/YAML, `hashutil` for hashing, `timeutil` for time, hermetic §14.6-style
tests, and the full `make ci` / `make ci-rehearse` gate discipline. Every
requirement below is normative unless marked *advisory*.

The deliverable, entirely within **overplane** (this module) plus the
documentation contract of §8:

1. Two new reusable packages: `internal/container` (engine-agnostic container
   client: docker and podman now; nerdctl and k3s stubbed for later, including
   asymmetric build/run engine pairs) and `internal/recipes` (typed registry
   of setup recipes, agent install recipes, and the runtime support matrix,
   loaded from an embedded, schema-validated YAML).
2. Embedded setup-script fragments imported and adapted from the proven
   aperture set (`/home/mayank/github/aperture/static/files`), parameterized
   by `overplane.yaml`, plus per-agent install fragments grounded in current
   official install instructions.
3. The `agent` top-level section of `overplane.yaml` (schema, loader,
   semantic + registry validation, init ripple).
4. The `agent` CLI command group: `overplane agent setup`,
   `overplane agent shell`, `overplane agent list-images` — uniform help
   throughout, themed output, correct exit codes.
5. Docs: the `agent` schema section renders into the existing Configuration
   reference; the CLI manifest gains the three subcommands; a **new**
   reference page ("Agent Environment") renders from the recipe registry.

These packages are **the core container abstraction of the entire project**:
the future `overplane agent run` (prompt execution, modeled on
`aperture/internal/harness/agent`) will be built on exactly these APIs.
Design for that consumer; do not specialize for `shell`.

---

## 1. Goal and non-goals

**Goal.** `overplane agent setup` builds (or cache-hits) a local,
project-labeled container image from `overplane.yaml`'s `agent.container`
section: a Debian (or Alpine) base prepared by embedded setup recipes — broad
language toolchains, z3, and developer conveniences — with the configured AI
coding agents (Codex, Claude Code, Gemini CLI) installed per current official
instructions. `overplane agent shell` drops the user into an ephemeral
interactive container that is environment-identical to what a future agent
run will see (same image, user identity, prompt, and passed-through API
keys). `overplane agent list-images` reports the project's built images.

**Non-goals.** No `agent run` / prompt execution (future spec; the
fileset-delta/isolated-workspace machinery of aperture's harness is **not**
ported here). No remote registries, pushing, or pulling beyond the base
image. No multi-platform / cross-arch builds (native platform only;
aperture's QEMU path is deliberately dropped for now). No Kubernetes. No
persistent agent vendor/home dirs (`~/.claude`, `~/.codex`, `~/.gemini`
persistence is deferred to the `agent run` spec). No project mounts in
`agent shell` (Appendix A item 4). No changes to the `check` command.

## 2. `overplane.yaml`: the `agent` section

### 2.1 Generated file (normative sample)

`overplane init` now generates this complete file (yamlcanon sorted order,
`MarshalPlainWithBanner`, same banner + pragma as #0002 §2.1; `project.name`
substitution unchanged):

```yaml
# Generated by overplane init. Safe to edit; re-run `overplane init` to validate.
# yaml-language-server: $schema=https://overplane.dev/schema/overplane.schema.json
agent:
  container:
    agent_recipes:
      - name: codex
        version: latest
      - name: claude-code
        version: latest
      - name: gemini-cli
        version: latest
    base_image: debian:bookworm-slim
    env_passthrough: []
    extra_packages:
      debian: []
    runtime: docker
    setup_recipe: debian
dirs:
  cache: .cache/overplane
  specs: .overplane/specs
model:
  default: ""
project:
  description: ""
  name: my-project
schema_version: 1
```

As in #0002, the schema's root `default` block is the single generation
source: extend it with the `agent` block above and `project.Default` picks it
up automatically. `schema_version` stays `1` — the new key is **optional**,
so every valid #0002-era file remains valid.

### 2.2 JSON Schema extension (embedded `overplane.schema.json`)

Extend the embedded schema (draft 2020-12, `additionalProperties: false` at
every object level, **elaborate user-facing `description` prose on every
field** — this renders verbatim on the public Configuration reference, §8.1).
New pointers:

| Pointer | Type | Constraints | Default |
|---|---|---|---|
| `/agent` | object | optional | (block in §2.1) |
| `/agent/container` | object | required within `agent` | — |
| `/agent/container/runtime` | string | enum `docker`, `podman` | `docker` |
| `/agent/container/base_image` | string | 1–200 chars; a Debian-like (or Alpine-like) image reference | `debian:bookworm-slim` |
| `/agent/container/setup_recipe` | string | enum `debian`, `alpine` | `debian` |
| `/agent/container/extra_packages` | object | optional; keys `debian`, `alpine` only | `{debian: []}` |
| `/agent/container/extra_packages/debian` | array of string | items match `^[a-z0-9][a-z0-9+.-]*$`; unique | `[]` |
| `/agent/container/extra_packages/alpine` | array of string | items match `^[a-z0-9][a-z0-9._+-]*$`; unique | — |
| `/agent/container/agent_recipes` | array of object | min 0 items; unique `name`s (semantic pass) | the three §2.1 entries |
| `/agent/container/agent_recipes/*/name` | string | enum `codex`, `claude-code`, `gemini-cli` (mirrors the registry, §3) | — |
| `/agent/container/agent_recipes/*/version` | string | 1–64 chars; an npm dist-tag or exact version (`latest`, `2.1.3`, `preview`); pattern `^[a-z0-9][a-z0-9._-]*$` | `latest` |
| `/agent/container/env_passthrough` | array of string | items match `^[A-Z_][A-Z0-9_]*$`; unique | `[]` |

Required within `/agent`: `container`. Required within `/agent/container`:
`runtime`, `base_image`, `setup_recipe`, `agent_recipes`.

**Schema-description intent (must be stated in the field prose):**

- `runtime` selects which engine builds **and** runs the agent container. The
  internal abstraction already models asymmetric build/run engine pairs
  (nerdctl-build/k3s-run, §5); exposing that in the file is deferred to a
  later `schema_version` (Appendix A item 2).
- `setup_recipe` names a built-in, embedded provisioning recipe (§3/§4) that
  must match the flavor of `base_image`; the recipe also determines how agent
  recipes are installed and which `extra_packages` key applies.
- `env_passthrough` lists **additional** host environment variables copied
  into the agent container; the variables each configured agent recipe needs
  (its API key) are declared in the registry and always passed through —
  this list is the project-specific union extension (Appendix A item 6).
  State plainly that live API keys reach the container (the docs warning).

### 2.3 Semantic + registry validation (third pass, Go)

`project.Config.Validate()` gains, reported as `jsonschema.Problem`s with
accurate pointers (e.g. `/agent/container/agent_recipes/1/name`):

- duplicate `agent_recipes[*].name` entries;
- duplicate entries within each `extra_packages.*` list and within
  `env_passthrough`.

A second, registry-aware pass `recipes.ValidateProjectConfig(reg, cfg)`
(§3.3) cross-checks the file against the embedded registry: `runtime` is a
supported run engine, `setup_recipe` exists, every `agent_recipes[*].name`
exists in the registry and is supported on the selected setup recipe, and
every `extra_packages` key names a package family the selected recipe (or any
registered recipe) defines. `project.Load` runs both passes, so any consumer
of a loaded config can trust the agent section completely.

### 2.4 Defaulting for pre-#0003 files

Files without an `agent` key stay untouched on disk (init's
non-destructiveness contract). `project.Load` **normalizes in memory**: after
decode, an absent `agent` block (or absent optional leaves with schema
defaults: `version`, `env_passthrough`, `extra_packages`) is filled from the
embedded schema's defaults so `cfg.Agent.Container` is always fully
populated. A unit test asserts a #0002-era golden file loads with the exact
§2.1 agent defaults.

### 2.5 Mirrors and drift

The #0002 §2.4 pattern applies unchanged: the embedded
`static/files/schema/overplane.schema.json` is canonical; the enclosing
repository's `config/schema/overplane.schema.json` mirror is enforced
byte-identical by the existing drift test.

- **Schema (canonical, embedded):** `static/files/schema/recipes.schema.json`

### 3.2 Content (normative shape; prose fields shown abridged)

subsystem supports. Every entry carries a `description` written as real
user-facing documentation (it is rendered on the public reference page).

### 3.3 Package: `internal/recipes`

App-specific package (may import `platform/*` and read the embedded assets;
`platform/*` and `internal/container` must not import it):

```go
type Registry struct { SchemaVersion int; ContainerRuntimes []Runtime; RuntimePairs []RuntimePair; SetupRecipes []SetupRecipe; AgentRecipes []AgentRecipe }

func Load() (*Registry, error)              // embedded recipes.yaml → schema validate (embedded
                                            // recipes.schema.json via configloader) → decode →
                                            // structural self-check (§3.4); cached after first call
func (r *Registry) SetupRecipe(name string) (SetupRecipe, bool)
func (r *Registry) AgentRecipe(name string) (AgentRecipe, bool)
func (r *Registry) EnvPassthrough(cfg project.AgentContainer) []string
                                            // union: per-recipe keys ∪ cfg.EnvPassthrough; sorted, deduped
func ValidateProjectConfig(r *Registry, cfg *project.Config) []jsonschema.Problem  // §2.3 second pass
func (r *Registry) BuildPlan(cfg project.AgentContainer, opts BuildPlanOptions) (*BuildPlan, error)
```

`BuildPlan` (consumed by the CLI, executed via `internal/container`) carries:
the ordered fragment list **with bytes** (setup fragments, then one agent
fragment per configured recipe in file order, then the user fragment, plus
entrypoint and profile assets), the generated Dockerfile text (§4.3), build
args, labels, image tags (§4.4), and the deterministic **build hash**:
SHA-256 short hash (`hashutil`) over base image ref, setup recipe name,
fragment names+bytes, Dockerfile text, sorted build args (which embed agent
names/versions and extra packages), and platform — the §4.4 cache key.

### 3.4 Structural self-check

`Load` fails (`InternalError`, exit 5 at the CLI boundary) unless: every
`fragments`/`user_fragment`/`install_fragment` name resolves to an embedded
`files/setup/<name>.sh` asset; every `runtime_pairs` entry references
declared runtimes with the matching capability; every
`supported_setup_recipes` entry exists; names are unique. A default-tier test
exercises this against the real embedded registry, so a bad registry can
never ship.

One fragment per registry agent recipe, taking the version as `$1`

| Fragment | Install command (current official instructions) |
|---|---|
| `agent-codex.sh` | `npm install -g @openai/codex@"$1"` — the official npm distribution of the Rust CLI (developers.openai.com/codex/cli) |

### 4.3 Generated Dockerfile (by `recipes.BuildPlan`)

Single-stage, generated in Go (no committed Dockerfile), layers ordered
slowest-changing first to maximize cache reuse, mirroring aperture's
`buildDockerfile` with these properties:

1. `ARG BASE_REF` / `FROM ${BASE_REF}`; args `SOURCE_DATE_EPOCH`,
   `EXTRA_OS_PACKAGES`, `OVERPLANE_UID/GID/USER/HOME`, `OVERPLANE_PROJECT`,
   toolchain version args, and one `<AGENT>_VERSION` arg per configured
   recipe.
2. `COPY` + `RUN` + `rm` per fragment in plan order: base-setup →
   install-deps → extra-packages → one layer per agent fragment (config file
   order) → user fragment. Each `RUN` deletes its fragment in the same layer.
3. `COPY bash_profile_extra.sh /etc/overplane/bash_profile_extra.sh`,
   `ENV OVERPLANE_PROJECT=...`, `COPY` + `chmod` the entrypoint,
   `ENTRYPOINT ["tini", "--", "/usr/local/bin/overplane-entrypoint.sh"]`.
4. Reproducibility: the build context is materialized in a temp dir with
   `SOURCE_DATE_EPOCH`-pinned mtimes; docker builds use `buildx` with
   `rewrite-timestamp=true`, podman uses
   `--source-date-epoch/--timestamp/--rewrite-timestamp` (port aperture's
   flags and its BuildKit ≥ 0.13 / Buildah ≥ 1.32 version gates with their
   actionable hints).

### 4.4 Naming, tagging, labeling (Appendix A item 1)

Every successful build produces:

- **Stable tag:** `overplane-<project.name>:latest` (always retagged to the
  current build).
- **Content tag:** `overplane-<project.name>:b<12-hex build hash>` (§3.3).
- **OCI labels:** `overplane.project=<project.name>`,
  `overplane.build.hash=<hash>`, `overplane.build.recipe=<setup_recipe>`,
  `overplane.build.base=<base_image>`,
  `overplane.version=<overplane CLI version>`.

The build-hash label is the cache key: `agent setup` first looks for an image
labeled with the computed hash (aperture's `cacheLookup`) and only builds on
a miss or `--force`.

## 5. Container engine abstraction: `internal/container`

App-agnostic-adjacent package (may import `platform/*` only — **not**
`internal/recipes` or `internal/project`; the CLI layer translates a
`BuildPlan` into the types below). Port aperture's
`internal/harness/container` design with overplane norms (logging via
`platform/log`, hashing via `hashutil`, themed dim streaming via
`platform/color`):

```go
type Engine int                      // EngineDocker, EnginePodman, EngineNerdctl, EngineK3s
type Capability int                  // CapBuild, CapRun
func ParseEngine(name string) (Engine, error)
func ValidatePair(build, run Engine) error    // §3.2 runtime_pairs matrix

type Client interface {
    Engine() Engine
    Capabilities() []Capability
    Available(ctx context.Context) error      // `<engine> version` probe + version gates (§4.3.4)
    BuildLocalImage(ctx context.Context, spec BuildSpec) (BuildResult, error)
    ListLocalImages(ctx context.Context, filter ImageFilter) ([]Image, error)
    RunLocalImage(ctx context.Context, ref Ref, args []string, opts RunOptions) (Container, error)
    ListLocalRunningContainers(ctx context.Context) ([]Container, error)
    KillLocalRunningContainer(ctx context.Context, id string) error
}

func New(ctx context.Context, buildEngine, runEngine Engine) (Client, error)
    // capability-routing multi-client; identical engines share one client
func NewSingle(engine Engine) (Client, error)
```

Normative points:

- **Docker and podman** are full CLI-driven implementations (subprocess
  `exec.CommandContext`, JSON `--format '{{json .}}'` parsing); **nerdctl and
  k3s are stubs** returning a sentinel `ErrEngineStub` with a hint — but the
  enum, capability routing, multi-client, and `ValidatePair` matrix are real,
  so adding an engine later touches only `newEngineClient`.
- `BuildSpec` takes a **prepared** context dir + Dockerfile path + tags +
  labels + sorted build args + build-hash (cache key) — context preparation
  is `internal/recipes`' job (§3.3); this package only drives the engine.
  This is a deliberate layering improvement over aperture, where the
  container package read embedded assets itself.
- `RunOptions` ports aperture's surface: env map, mounts, network mode, user
  spec (uid:gid), workdir, stdin/stdout/stderr, TTY, detach, name, labels.
  Interactive runs (stdin or TTY set) use a **streaming** runner that leaves
  the child in the foreground process group; non-interactive runs use the
  buffered runner with line-streamed output. `pull`/`build` subcommand output
  streams to the terminal **dim** (themed via `color`, plain under
  `NO_COLOR`), matching aperture's UX.
- The exec boundary is an injectable `commandRunner` interface (package var
  or constructor arg) so the entire package is testable with a fake runner —
  no real daemon in the default test tier.
- Sentinel errors (`ErrDaemonUnreachable`, `ErrBuildFailed`, `ErrRunFailed`,
  `ErrEngineStub`, `ErrUnsupportedCapability`, ...) each carry a
  `Hint() string` with actionable remediation (port aperture's hints,
  including the buildx/binfmt/auth/apt build-failure classifier).
- Context cancellation must terminate the engine subprocess group (port
  `setProcessGroup`/`terminateProcessGroup` + `WaitDelay`), and a timed-out
  named container is best-effort `stop`/`kill`ed.

## 6. CLI: the `agent` command group

### 6.1 Group surface

| Item | Value |
|---|---|
| Registry entry | `agent` — "manage the project's agent container" |
| Usage | `overplane agent <subcommand> [flags]` |
| Subcommands | `setup`, `shell`, `list-images` |
| Routing | `runSubcommandGroup` (like `config`/`theme`), `missingIsError: true` |
| Help | group `HelpSpec` listing the three subcommands + per-subcommand `HelpSpec`s with flags and ≥ 2 deterministic examples each; every level answers `help`, `-h`, `--help` — **uniform with the rest of the CLI** |
| Kind | config-optional interface, **config-required semantics**: each subcommand calls `project.Locate` + `project.Load` first and fails with exit 6 and hint `run 'overplane init' first` when no `overplane.yaml` is found (validation failures exit 3 as usual). This is the first config-required command family; per #0002 §4.1 the dispatcher-level `ConfigRequiredCommand` interface is **still not introduced** — factor a small shared `requireProject(ctx)` helper inside `internal/cli` instead, and note in `AGENTS.md` that a third consumer should trigger the interface |

### 6.2 `overplane agent setup [--force]`

Phases, in order, stopping at first failure:

1. **Load + validate** project config (§2.3/§2.4) and registry (§3.3).
2. **Engine availability** — `container.New` for the configured runtime;
   `Available(ctx)` including version gates. Failure → exit 6 with hint.
3. **Plan** — `recipes.BuildPlan` (fragments, Dockerfile, args incl. host
   uid/gid/user/home and `OVERPLANE_PROJECT`, tags, labels, build hash).
   INFO-log the plan summary (`step=agent-setup-plan`: recipe, base, agents,
   hash).
4. **Cache lookup** — image with label `overplane.build.hash=<hash>` exists
   and `--force` not set → retag `:latest` if needed, INFO
   `step=agent-setup-cache-hit`, skip to 6.
5. **Build** — materialize context (temp dir, cleaned up), stream engine
   output dim, on success INFO `step=agent-setup-built` with image id, size,
   duration. `--force` passes the engine's no-cache flag so `latest` agent
   versions are re-resolved (Appendix A item 8). Build failure → exit 1 with
   the classified hint.
6. **Summary** — themed table on stdout: image tags, id (short), size,
   created, recipe, base, agents (name@version), build hash; then a
   next-steps line suggesting `overplane agent shell`.

Re-running with an unchanged config is a fast, no-build cache hit (a test
enforces: second run performs no `build` engine invocation).

### 6.3 `overplane agent shell`

Spawns the project's built image as an **ephemeral interactive container**,
as close as possible to the environment a real agent run will get — minus
workspace mounts (Appendix A items 4–5):

- Preconditions: a loaded project config; interactive TTY
  (`tui.EnsureInteractive`-style guard → exit 6 with hint in a pipe); image
  `overplane-<project.name>:latest` present → else exit 6, hint
  `run 'overplane agent setup' first`.
- Run: `--rm`, interactive TTY (`-it`), `NetworkMode: host`, user =
  host uid:gid (`container.UserSpec`), workdir = the baked
  `OVERPLANE_HOME`, name `overplane-shell-<project.name>-<unix-ts>`, labels
  `overplane.project=<name>` + `overplane.shell=true`. **No mounts of any
  kind** — everything inside is discarded on exit (matches the public docs'
  promise).
- Command: `bash -l` (login shell so `/etc/profile.d` toolchain paths and the
  branded prompt load). The entrypoint's no-`/workspace-ro` fall-through
  (§4.1) makes this work without mounts.
- **Env passthrough:** `registry.EnvPassthrough(cfg)` (recipe API keys ∪
  `agent.container.env_passthrough`) resolved via `platform/env.Passthrough`
  (#0001 §17 — this is its first consumer; values whitespace-trimmed), plus
  `TERM` and `LANG`. Host `PATH`/`HOME` are **not** forwarded (container owns
  them). A configured recipe whose key is unset on the host logs one WARN
  (`step=agent-shell-env`, `hint=export <KEY> to use <agent> inside the
  container`) and continues — the shell is also for inspecting the image.
- Exit: the container's exit status becomes the command's exit status
  (a normal `exit` is 0); INFO `step=agent-shell-exit` with duration.

### 6.4 `overplane agent list-images [--json]`

Lists **this project's** images only (Appendix A item 7): engine image list
filtered by label `overplane.project=<project.name>`. Output: themed table
(Tag, Image ID (short), Size, Created (UTC RFC3339 via `timeutil`), Recipe,
Base, Build hash; the `:latest`-tagged row first); `--json` emits the same
records as a canonical JSON array (`canonjson.MarshalIndent`). Zero images is
**success** (exit 0) with an INFO line + hint to run `agent setup`; engine
unavailability is exit 6.

### 6.5 Exit codes (per the #0001 table)

| Condition | Code |
|---|---|
| success | 0 |
| engine build/run failure (daemon reachable) | 1 |
| usage error (unknown subcommand, stray args, bad flag) | 2 |
| `overplane.yaml` schema/semantic/registry validation failure | 3 |
| filesystem/IO failure (build context temp dir, ...) | 4 |
| embedded registry invalid / fragment missing (internal) | 5 |
| no project file; engine unavailable/too old; image missing for shell; non-TTY shell | 6 |

### 6.6 Logging

#0002 §5 presentation contract applies: INFO for one-line phase summaries,
DEBUG for decision detail (cache probe, env key resolution, full engine
argv), WARN only for things the user should notice (missing API key for a
configured agent, `--force` discarding a cached image). Steps:
`agent-setup-plan`, `agent-setup-cache-hit`, `agent-setup-build`,
`agent-setup-built`, `agent-shell-env`, `agent-shell-start`,
`agent-shell-exit`, `agent-list-images`. Engine stdout/stderr during
build/pull is dim pass-through (not slog records); secrets (API key values)
must never appear in any log line — log key **names** and presence booleans
only (aperture's preflight pattern).

## 11. Acceptance criteria

1. `make ci` and `make ci-rehearse` pass; the new `e2e` tier is excluded from
   both and runs green via `make e2e` on a machine with docker.
2. In a fresh initialized project, `overplane agent setup` builds an image
   tagged `overplane-<name>:latest` + `overplane-<name>:b<hash>` carrying all
   §4.4 labels; an immediate re-run is a cache hit performing **no** engine
   build; `--force` rebuilds with no-cache.
3. The built debian image contains the §4.1 toolchain set (go, rustc/cargo,
   node ≥ the NodeSource major, uv, z3, ripgrep, build-essential, git, tini,
   sudo) and the three configured agent CLIs at their requested versions;
   `graphicsmagick` is absent by default and present when listed in
   `extra_packages.debian`; the `alpine` recipe builds and runs the three
   agent CLIs.
4. `overplane agent shell` (in a TTY) lands in a login bash with the
   `⌬ overplane:<project>` prompt, host uid/gid, host networking, **no
   mounts**, and the recipe + project passthrough env vars (trimmed) set;
   files created inside are gone after exit; in a pipe it exits 6 with the
   interactivity hint; with no built image it exits 6 with the setup hint.
5. `overplane agent list-images` lists only this project's images (table and
   canonical `--json`), exits 0 with a hint when none exist.
6. With no `overplane.yaml`, every `agent` subcommand exits 6 with the init
   hint; with a hand-broken agent section (unknown agent name, unsupported
   runtime, bad package name) it exits 3 listing pointer-accurate problems;
   with no working engine it exits 6.
7. A #0002-era `overplane.yaml` without the `agent` key keeps working
   unmodified (in-memory defaults); a fresh `overplane init` writes the §2.1
   document byte-exactly (modulo `project.name`).
8. `overplane config validate path/to/recipes.yaml` exits 0 on the shipped
   registry and 3 on a corrupted copy.
9. The embedded `recipes.yaml`/`recipes.schema.json`/`overplane.schema.json`
   are byte-identical to the enclosing-repo mirrors (drift tests green).
10. Help is uniform: `overplane agent`, each subcommand's `--help`, and the
    root help entry all render through `HelpSpec`/`clihelp` with examples;
    unknown subcommands exit 2 with usage.
11. The public Configuration reference renders the elaborate `agent` section
    prose; the new Agent Environment reference page renders runtimes,
    setup recipes, and agent recipes from the registry; the Command Line
    reference lists the three subcommands directly after `init` (§8).
12. No log line ever contains an API key value; engine build output streams
    dim (plain under `NO_COLOR`).

## Appendix A. Clarification answers

Supplied before writing this spec; normative.

1. Image identity: stable tag `overplane-<project.name>:latest` **plus** a
   content-hash tag `overplane-<project.name>:b<12-hex>` **plus** OCI labels
   (`overplane.project`, `overplane.build.hash`, `overplane.build.recipe`,
   `overplane.build.base`).
2. `overplane.yaml` exposes a single string `agent.container.runtime`
   (`docker`|`podman`); the Go abstraction models asymmetric build/run engine
   pairs internally, with schema exposure deferred to a later
   `schema_version`.
3. The recipe registry renders as a **new reference page** (Agent
   Environment) generated from the typed YAML, added to the reference nav —
   not a section of the Configuration page.
4. `agent shell` mounts **nothing**: a bare container exposing the image's
   toolchains and agents; the rsync workspace machinery ships in the image
   for the future `agent run` but is bypassed.
5. No persistent vendor/home dir mounts (`~/.claude`, `~/.codex`,
   `~/.gemini`) in this spec; persistence is deferred to the `agent run`
   spec. Shell containers are fully ephemeral.
6. `overplane.yaml` gains an optional `agent.container.env_passthrough`
   list, unioned with the registry-declared per-recipe keys.
7. `agent list-images` shows only the **current project's** images.
8. `agent setup` uses an aperture-style content-hash label cache (skip
   rebuild on hash match) plus a `--force` flag that rebuilds with the
   engine's no-cache mode, re-resolving `latest` agent versions.
9. Package layout: `internal/container` (engine client) + `internal/recipes`
   (typed registry + build planning) + `internal/cli/agentcmd.go` wiring —
   not a single monolithic package.
10. The alpine setup recipe targets best-effort toolchain parity with debian
    via apk, with documented gaps allowed; all three agent CLIs must work on
    it (musl builds exist for each).
11. Claude Code installs via `npm install -g @anthropic-ai/claude-code`
    (which installs the official native binary), not the per-user
    `claude.ai/install.sh` script — uniform with codex/gemini, system-wide,
    and version-pinnable in a container layer.

## Appendix B. As-built record

Implemented as specified, with the following notes and deviations:

1. **Registry/matrix lockstep test** lives in `internal/recipes`
   (`TestRuntimePairsMatchContainerMatrix`) rather than `internal/container`,
   because `recipes` may import `container` but not vice versa (the container
   package stays app-agnostic). Both directions are asserted: every registry
   pair is allowed by `ValidatePair` and every allowed pair is declared.
2. **`yamlcanon` grew two rendering refinements** to make the §2.1 default
   document byte-exact: block sequences of mappings render compactly
   (`- name: codex` on the dash line) and `:` no longer forces quoting in
   non-edge positions (`debian:bookworm-slim` renders plain). Covered by
   `TestMarshalPlainBlockSequences`.
3. **Generated docs manifest** is named `recipes.overplane.json` (rendered by
   `sitebuild` from `config/data/recipes.yaml` via the new
   `config.LoadRecipes` + `gen_recipes.go` go-jsonschema models); the Agent
   Environment page and the reference nav consume it. The nav entry sits
   between Command Line and Spec.
4. **e2e tier** is binary-exec (`cmd/overplane/e2e_test.go`,
   `//go:build e2e`), run by `make e2e`, auto-skipping without a daemon.
   Verified green against real Docker (buildx v0.34): fresh build ~3.5 min
   warm-network, cache-hit re-run sub-minute, list-images labels asserted,
   and the in-container probe confirmed codex/claude/gemini `--version`, z3,
   go, rustc, node, and the absence of `gm`.
5. **§8.4 screencasts are not recorded** (advisory item; requires a VHS
   recording pass). The getting-started prose was reconciled with as-built
   behavior instead: the `agent.container` example now shows the real
   generated shape (project-level `env_passthrough`, no per-agent key lists,
   canonical field order) and the Gemini key is `GEMINI_API_KEY`.
6. **`agent shell` exit-status passthrough** maps the container's exit code
   directly onto the CLI exit code (per §6.3), even where it collides with
   the §6.5 table values; the error line says `shell exited with status N`.
7. **`make ci` caveats**: `gate-generate` and `gate-fmt` diff the working
   tree against the git index, so they pass only on a committed tree; both
   were verified idempotent/clean manually. The coverage gate is
   informational (reports total; no floor enforcement) as it was before this
   spec.
